## Context

The converged decomposition plan (`swift-institute/Research/dual-defunctionalize-decomposition.md`) established three independent macros with distinct theoretical contracts:

| Macro | Input | Output | Domain |
|-------|-------|--------|--------|
| `@Dual` | Struct or enum | Structural dual | Category theory |
| `@Defunctionalize` | Struct with function-typed properties | Call algebra enum | Reynolds 1972 |
| `@Witness` | Struct with function-typed properties | DI/test/observation scaffolding | Software engineering |

The converged plan's contracts table states `@Witness` input is "Struct with function-typed stored properties" -- no mention of enum input. `@Defunctionalize` on enums is explicitly an error.

Meanwhile, `@Witness` currently accepts enums and generates prism infrastructure (extraction properties, `Case` discriminant, `Prisms` struct, `is(_:)`, `subscript[prism:]`, `modify(_:_:)`). The literature survey (`swift-witnesses/Research/action-type-naming.md`) explicitly noted: "the current enum expansion is NOT a true dual" -- it generates prism access infrastructure, not the categorical dual.

`@Dual` on enums now generates a strict superset of what `@Witness` generates: Scott encoding (`Dual<R>` struct + `match`), plus all the same enum infrastructure (extraction properties, `Case` discriminant, `Prisms`, `is(_:)`, `subscript[prism:]`, `modify(_:_:)`). The infrastructure is generated by shared codegen functions (`ExtractionCodegen.swift`, `CaseDiscriminantCodegen.swift`, `PrismCodegen.swift`).

**Trigger**: [RES-001] Architecture decision -- macro input domain overlap creates redundancy and semantic confusion.

## Question

Should `@Witness` on enums be removed now that `@Dual` exists? If not, what is the correct relationship between `@Witness` on enums and `@Dual` on enums?

## Analysis

### Option A: Remove enum support from @Witness entirely

Users apply `@Dual` to enums for enum infrastructure. `@Witness` becomes struct-only, matching the converged plan's contracts table.

**Advantages**:

1. **Converged plan compliance.** The contracts table says `@Witness` input is "Struct with function-typed stored properties." Enum support was never part of the converged contract.

2. **No API surface redundancy.** Today, `@Witness` on an enum and `@Dual` on an enum generate overlapping output. `@Dual` generates a strict superset (adds Scott encoding + `match`). Maintaining two macros that generate the same infrastructure for the same input type is unjustified duplication.

3. **Semantic clarity.** `@Witness` on an enum has no clear conceptual basis. A witness is a struct-of-closures representing a capability (per the handler-witness identity analysis in `swift-institute/Research/handler-witness-conceptual-identity.md`). An enum is not a witness -- it is a sum type. The enum infrastructure (prisms, discriminants, extraction) is cross-cutting codegen convention, not part of the witness concept.

4. **Clean input domain partitioning.** Each macro accepts exactly one kind of input:
   - `@Dual`: struct or enum (structural dual in both directions)
   - `@Defunctionalize`: struct with closures (call algebra extraction)
   - `@Witness`: struct with closures (DI composition)

   No overlap on enums.

**Disadvantages**:

1. **Migration cost.** Existing `@Witness` enum users must change to `@Dual`. However, the migration cost is near-zero: the only production uses of `@Witness` on enums found in the codebase are test fixtures within `swift-witnesses/Tests/` itself. No production code in swift-primitives, swift-standards, or swift-foundations (outside swift-witnesses) uses `@Witness` on an enum.

2. **Users who want only infrastructure (not Scott encoding) get more than they need.** With `@Dual`, they also get `Dual<R>` and `match`, which they may not use. This is a minor concern -- unused generated code has no runtime cost and minimal compile-time cost.

### Option B: @Witness on enums internally applies @Dual

`@Witness` on an enum delegates to `@Dual` and generates the full dual including Scott encoding, `match`, and all infrastructure.

**Advantages**:

1. Backwards compatible -- `@Witness` still compiles on enums.
2. Users get the full dual without knowing about `@Dual`.

**Disadvantages**:

1. **Violates the converged plan.** `@Witness` was explicitly scoped to struct input. Making it silently delegate to `@Dual` introduces a hidden dependency and blurs the boundary between independent macros.

2. **Semantic confusion.** `@Witness` on an enum generating a `Dual<R>` struct is incoherent. The witness concept is about DI composition (observe, unimplemented, mock). None of those concepts apply to a raw enum with case data.

3. **Creates a false equivalence.** Users would think `@Witness` and `@Dual` are interchangeable on enums, when they have fundamentally different purposes.

4. **Dependency direction.** `swift-witnesses` would need to depend on `swift-dual` or duplicate its codegen. The converged architecture has them as independent packages importing shared codegen from a neutral support package. Making one delegate to the other introduces coupling.

### Option C: @Witness on enums delegates to the same codegen as @Dual but omits Scott encoding

`@Witness` on an enum generates only the infrastructure subset (extraction, Case, Prisms, accessors) -- the same output as today -- but using shared codegen from `swift-codegen-support`.

**Advantages**:

1. Backwards compatible.
2. No redundant Scott encoding generation.
3. Shared codegen reduces maintenance.

**Disadvantages**:

1. **Still violates the converged plan.** `@Witness` input domain includes enums, which the contracts table does not specify.

2. **Redundant with `@Dual`.** `@Dual` on the same enum generates everything `@Witness` generates plus more. Two macros producing overlapping output on the same input type is unjustified.

3. **"Infrastructure-only" is not a coherent concept for `@Witness`.** The enum infrastructure (prisms, discriminants) is a cross-cutting codegen convention. It is not specific to witnesses. The converged plan explicitly classifies it as such: "Enum infrastructure (prisms, discriminants, extraction) is cross-cutting codegen convention, not part of either theory's contract. Any macro-generated enum receives standard utilities." The convention is that ANY macro that generates an enum attaches this infrastructure. That is `@Dual`'s job for user-authored enums.

4. **Semantic muddle persists.** Users still have to decide between `@Witness` and `@Dual` for enums, with no clear reason to prefer one.

### Option D: Keep current behavior unchanged

Both `@Witness` and `@Dual` accept enums. `@Witness` generates the infrastructure subset. `@Dual` generates the infrastructure plus Scott encoding.

**Advantages**:

1. Zero migration cost.
2. Users who only want infrastructure can use `@Witness`.

**Disadvantages**:

1. **Directly contradicts the converged plan.** The contracts table was agreed upon by both Claude and ChatGPT in Round 4 of the collaborative discussion. Enum input for `@Witness` was deliberately excluded.

2. **Permanent API surface redundancy.** Two macros generating overlapping code for the same input type is a maintenance burden and a source of confusion.

3. **The "infrastructure-only" use case is not compelling.** The incremental cost of `@Dual` generating `Dual<R>` and `match` beyond the infrastructure is negligible (unused code, no runtime cost). There is no scenario where a user benefits from having prism infrastructure but NOT the Scott encoding.

4. **Blocks `@Witness` refactoring.** The converged plan calls for refactoring `@Witness` to compose on `@Defunctionalize`. Retaining enum support complicates this refactoring because enum expansion has nothing to do with defunctionalization.

### Comparison

| Criterion | A: Remove | B: Delegate to @Dual | C: Shared codegen subset | D: Keep unchanged |
|-----------|-----------|---------------------|--------------------------|-------------------|
| Converged plan compliance | Full | Violates | Violates | Violates |
| API surface redundancy | None | Worse (hidden delegation) | Same as today | Same as today |
| Semantic clarity | High | Low (witness = dual?) | Medium | Low |
| User migration cost | Near-zero | None | None | None |
| Backwards compatibility | Breaking (test fixtures only) | Full | Full | Full |
| Maintenance burden | Reduced | Increased (coupling) | Same | Same |
| Blocks @Witness refactoring | No | Yes | Yes | Yes |

## Outcome

**Status**: RECOMMENDATION

**Option A: Remove enum support from @Witness entirely.**

**Rationale**:

1. **The converged plan already decided this.** The contracts table explicitly scopes `@Witness` to "Struct with function-typed stored properties." Enum support was never part of the agreed contract. Removing it is not a new decision -- it is implementing the existing decision.

2. **@Dual is the correct macro for enums.** `@Dual` on enums generates the categorical dual (Scott encoding) plus all infrastructure. This is semantically precise: the dual of a coproduct (enum) is a product of handlers (struct). `@Witness` on an enum has no corresponding theoretical basis.

3. **Migration cost is effectively zero.** The only uses of `@Witness` on enums in the entire codebase are two test fixture enums in `swift-witnesses/Tests/Witnesses Tests/Witness.Enum Tests.swift`. No production code anywhere uses this feature. The migration is: change `@Witness` to `@Dual` on those two enums, update test imports.

4. **Removal simplifies the @Witness refactoring.** The converged plan calls for refactoring `@Witness` to compose on `@Defunctionalize`. With enum support removed, `@Witness` is purely struct-focused, which aligns with the defunctionalization pipeline (struct → call algebra → DI scaffolding).

5. **The "infrastructure subset" argument does not hold.** The extra output from `@Dual` (Scott encoding + `match`) has no runtime cost and minimal compile-time cost. There is no user who benefits from getting prism infrastructure without also getting the Scott encoding. If anything, the Scott encoding is the primary value of `@Dual` on enums -- the infrastructure is a bonus.

**Implementation path**:

1. In `WitnessMacro.swift`, change the enum branch to emit a diagnostic error: "@Witness cannot be applied to enums. Use @Dual instead."
2. Update `Witness.Enum Tests.swift` to use `@Dual` (or move those tests to swift-dual).
3. Remove `EnumExpansion.swift` from `Witnesses Macros Implementation/` (the shared codegen functions in swift-dual's `ExtractionCodegen.swift`, `CaseDiscriminantCodegen.swift`, `PrismCodegen.swift` are the canonical implementations).
