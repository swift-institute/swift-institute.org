## Context

The Swift Institute ecosystem includes five packages concerned with struct-of-closures patterns and enum infrastructure, inspired by PointFree's work:

| Our Package | PointFree Equivalent | Layer | Current State |
|-------------|---------------------|-------|---------------|
| `swift-dual` | `swift-case-paths` (generalized) | L3 | Implemented |
| `swift-defunctionalize` | (no equivalent) | L3 | Implemented |
| `swift-witnesses` | `swift-dependencies` `@DependencyClient` | L3 | Implemented |
| `swift-dependencies` | `swift-dependencies` container | L3 | Implemented |
| `swift-dependency-primitives` | (no equivalent) | L1 | Implemented |

The original insight was that PointFree's `@CasePathable` (enum case extraction via prisms) could be generalized to both enums AND structs -- the categorical dual. This was initially embedded in `@Witness`, then decomposed into `@Dual` / `@Defunctionalize` / `@Witness` via a Claude+ChatGPT collaborative discussion (`dual-defunctionalize-decomposition.md`).

### Trigger

[RES-001] After the decomposition, the boundaries between the three macros have become unclear. Three independent packages duplicate identical enum infrastructure codegen. The user has lost overview of how these should relate and compose. This document establishes the definitive architectural relationship.

### Scope

Ecosystem-wide per [RES-002a]. The decisions here govern the relationship between five packages across two layers and establish the macro composition pattern for the ecosystem.

---

## Question

**Primary**: What is the correct scope, relationship, and composition of `@Dual`, `@Defunctionalize`, `@Witness`, `swift-dependencies`, and `swift-dependency-primitives`? How should they relate to eliminate functionality duplication while keeping each concept clean?

**Secondary**: Should users ever need to apply more than one of these macros to a single type? Can Swift macros compose (can one macro's expansion see another's)?

## Prior Art Survey [RES-021]

### PointFree's Architecture

PointFree provides exactly two user-facing macro packages for this domain:

| Package | Macro | What it does |
|---------|-------|-------------|
| `swift-case-paths` | `@CasePathable` | Enum infrastructure: case key paths (analogous to prisms), extraction, pattern matching, `is(_:)`, `modify(_:_:)` |
| `swift-dependencies` | `@DependencyClient` | Witness struct infrastructure: Calls enum ("call algebra"), `unimplemented()`, convenience methods with argument labels, public init |

Key architectural decisions in PointFree's ecosystem:

1. **`@CasePathable` is enum-only.** It does not operate on structs. The struct direction (product -> coproduct) was not pursued.

2. **`@DependencyClient` is struct-only.** It generates its own Calls enum internally. It does NOT depend on or delegate to `@CasePathable`. The Calls enum it generates does NOT get `@CasePathable` infrastructure (no case key paths on Calls).

3. **No macro composition.** `@DependencyClient` generates everything it needs independently. Users never stack `@CasePathable` + `@DependencyClient`. Each macro is self-contained.

4. **The DI container is separate.** `swift-dependencies` provides `@Dependency` property wrapper, `withDependencies`, `DependencyValues`, `TestDependencyKey`. It consumes `@DependencyClient`-annotated types but does not generate code.

5. **`@CasePathable` lives in `swift-case-paths`** (the optics package). `@DependencyClient` lives in `swift-dependencies` (the DI package). They are in separate repositories with no dependency between them.

**Implication for our ecosystem**: PointFree accepted the duplication. `@DependencyClient` generates a simple Calls enum without prism infrastructure. `@CasePathable` generates rich enum infrastructure. They overlap conceptually but neither depends on the other. The simplicity of two independent, non-composing macros is valued over eliminating ~100 lines of shared codegen.

### Swift Macro Composition Constraints

Swift macros have a fundamental limitation relevant to this analysis:

1. **Macros cannot see other macros' expansions.** When the compiler expands `@Witness`, it operates on the original source. It cannot see members generated by `@Dual` or `@Defunctionalize`, even if those macros are also attached to the same type. Each macro expansion is independent.

2. **Macros can check for other attributes.** A macro implementation CAN check whether `@Dual` is present on the declaration (by inspecting `declaration.attributes`). But it cannot access what `@Dual` would generate.

3. **No macro-applies-macro.** A macro expansion cannot programmatically apply another macro to a generated type. `@Witness` cannot say "and now apply `@Dual` to the Calls enum I just generated."

**Consequence**: Options that rely on `@Witness` "delegating to" `@Dual` at the macro expansion level are technically impossible in Swift. The only viable forms of composition are:
- **Shared codegen library**: multiple macro implementations import the same Swift code for generating enum infrastructure.
- **User-level stacking**: users apply multiple macros, each generating independent output that happens to compose at the type level (but neither macro sees the other's output).
- **Subsumption**: one macro generates everything, including what another macro would generate.

### TCA (The Composable Architecture)

TCA uses a single `@Reducer` macro that generates all needed infrastructure. It does not compose with `@CasePathable` -- instead, types that need case paths apply `@CasePathable` separately. The user stacks macros when both are needed:

```swift
@Reducer
struct Feature {
    @CasePathable
    enum Action { ... }
}
```

This works because `@CasePathable` operates on the user-written `enum Action`, not on anything `@Reducer` generates. The two macros do not interact at the expansion level.

### Rust Ecosystem

Rust's proc-macro system has the same constraint: derive macros cannot see other derive macros' output. The Rust ecosystem handles shared codegen via regular library crates (`serde_derive_internals`, `syn`, `quote`). Multiple proc-macro crates import shared codegen from these libraries.

## Current State Analysis

### What each macro generates

To evaluate composition options, here is a precise inventory of what each macro generates:

**@Dual on a struct** (e.g., `struct Config { var host: String; var port: Int }`):
- `Config.Dual` enum with cases `host(String)`, `port(Int)`
- Extraction properties (`var host: String?`, `var port: Int?`)
- `Case: Finite.Enumerable` discriminant enum
- `var case: Case` computed property
- `Prisms` struct with `Optic.Prism` per case
- `static var prisms`, `is(_:)`, `subscript[prism:]`, `modify(_:_:)`
- `__OpticPrismAccessible` conformance (on the Dual enum)
- Homogeneous `subscript(case:)` on the source struct (when all fields share a type)

**@Dual on an enum** (e.g., `enum Route { case home; case profile(id: Int) }`):
- `Route.Dual<R>` struct (Scott encoding: one closure per case, returning R)
- `match(_:)` function for case analysis
- Extraction properties on the original enum
- `Case: Finite.Enumerable` discriminant enum
- `var case: Case`
- `Prisms` struct
- `static var prisms`, `is(_:)`, `subscript[prism:]`, `modify(_:_:)`
- `__OpticPrismAccessible` conformance

**@Defunctionalize on a struct** (e.g., `struct Service { var fetch: (_ id: Int) -> String; var timeout: Duration }`):
- `Service.Calls` enum with cases derived from function-typed properties only (`case fetch(id: Int)`)
- Non-function fields excluded
- Ownership-annotated parameters excluded from cases
- Extraction properties on the Calls enum
- `Case: Finite.Enumerable` discriminant enum
- `var case: Case`
- `Prisms` struct
- `static var prisms`, `is(_:)`, `subscript[prism:]`, `modify(_:_:)`
- `__OpticPrismAccessible` conformance

**@Witness on a struct** (e.g., `struct APIClient: Sendable { var fetch: (_ id: Int) async throws -> User }`):
- Public memberwise initializer (if public, no existing init)
- Convenience methods with argument labels for labeled closures
- `APIClient.Calls` enum (same as @Defunctionalize: function-typed properties only)
- Extraction properties, Case discriminant, Prisms, prism accessors on Calls
- `Observe` struct for wrapping with observation closures
- `var observe: Observe` accessor
- `static func unimplemented(...)` returning a total placeholder
- `static func mock(...)` (if `.mock` derive option)
- `callAsFunction()` + `static func constant(_:)` (if `.generator` derive option)
- `_Witness` typealias
- `__WitnessProtocol` conformance
- `__OpticPrismAccessible` conformance

**@Witness on an enum** (recommended for removal per `witness-enum-direction-and-dual.md`):
- Same enum infrastructure as @Dual on an enum, minus Scott encoding

## Options Analysis

### Option A: Current State -- Three Independent Macros with Duplicated Codegen

Maintain `@Dual`, `@Defunctionalize`, and `@Witness` as fully independent packages. Each contains its own copy of enum infrastructure codegen. No dependency between them.

**User-facing surface**: Three macros to learn.
- `@Dual` for structural duality (both structs and enums)
- `@Defunctionalize` for call algebra extraction
- `@Witness` for DI witness structs

**When to apply which**:
- Plain enum with rich infrastructure: `@Dual`
- Struct questionnaire with Dual enum: `@Dual`
- Struct with closures, call algebra only: `@Defunctionalize`
- Struct with closures, full DI: `@Witness`

**Advantages**:
1. Zero coupling between packages. Any can evolve independently.
2. No dependency graph complexity. Each is self-contained.
3. Follows PointFree's precedent: independent, non-composing macros.
4. No coordination overhead for releases or breaking changes.

**Disadvantages**:
1. ~280 lines of identical codegen in three places. When `Finite.Enumerable` requirements change or `Optic.Prism` API evolves, three files must be updated identically.
2. Three macros for what could be two concepts (structural vs DI).
3. `@Defunctionalize` and `@Witness` generate overlapping Calls enums. A user who wants call algebra + DI must choose @Witness (which subsumes @Defunctionalize). There is no additive composition.
4. Contradicts the converged plan's statement: "Shared codegen lives in a neutral support package."

---

### Option B: @Witness Depends on @Dual/@Defunctionalize at the Package Level

`swift-witnesses` imports `swift-dual` and/or `swift-defunctionalize`, reusing their codegen at the macro implementation level. Users still apply only `@Witness`.

**Advantages**:
1. Single source of truth for shared codegen.
2. User-facing simplicity unchanged.

**Disadvantages**:
1. **Impossible at the macro level.** Swift macro targets are compiler plugin executables. A `.macro` target can import regular library targets, but it cannot import another `.macro` target's internal symbols. `swift-witnesses`'s macro cannot import `swift-dual`'s macro.
2. Even if sharing at the regular library level (not macro level), `swift-witnesses` depending on `swift-dual` creates a false semantic edge: DI witnesses do not depend on categorical duality.
3. Version coupling: changes to `swift-dual`'s codegen would require `swift-witnesses` to update.

**Verdict**: Technically constrained and semantically wrong. Eliminated.

---

### Option C: @Witness Applies @Dual Internally

When `@Witness` expands a struct, it programmatically applies `@Dual` to the generated Calls enum to get enum infrastructure.

**Verdict**: Eliminated. Swift macros cannot apply other macros. The compiler does not support macro-applies-macro. See Prior Art Survey above.

---

### Option D: Shared Codegen Library (swift-enum-infrastructure-codegen)

Extract `PrismCodegen.swift`, `CaseDiscriminantCodegen.swift`, `ExtractionCodegen.swift`, and `Utilities.swift` into a shared Layer 3 library package. All three macro implementation targets import this library.

This is the recommendation from `enum-infrastructure-primitives.md` (Option F in that document).

**Structure**:
```
swift-enum-infrastructure-codegen/     Layer 3, SwiftSyntax-dependent
├── Package.swift
└── Sources/
    └── Enum Infrastructure Codegen/
        ├── PrismCodegen.swift
        ├── CaseDiscriminantCodegen.swift
        ├── ExtractionCodegen.swift
        └── Utilities.swift
```

**Consumer integration**: Each macro's `.macro` target adds a dependency on `Enum Infrastructure Codegen`.

**Advantages**:
1. Single source of truth for enum infrastructure codegen. One place to update.
2. Follows the Rust `serde_derive_internals` precedent.
3. No false semantic edges. The dependency accurately says: "this macro uses shared enum codegen."
4. Fulfills the converged plan's requirement.

**Disadvantages**:
1. New package to version and maintain.
2. "Enum infrastructure codegen" is a process name, not a domain name. Naming is mechanical rather than elegant.
3. Build graph now has an extra node. Minor compile-time cost.
4. Coordination: all three consumers must align on the same version of the shared library.

---

### Option E: Users Stack @Defunctionalize + @Witness

`@Witness` generates ONLY the DI additions (methods, observe, unimplemented, mock). Users apply `@Defunctionalize` first to generate the Calls enum, then `@Witness` adds DI scaffolding on top.

**Verdict**: Eliminated. Swift macros cannot see other macros' expansions. `@Witness` would not be able to reference the `Calls` enum generated by `@Defunctionalize`. The expansion would fail with "cannot find type 'Calls' in scope."

---

### Option F: @Witness Subsumes @Defunctionalize

`@Witness` IS the DI-augmented defunctionalization. `@Defunctionalize` exists only for the pure academic use case where someone wants call algebra without DI infrastructure. `@Dual` is the independent structural primitive. Two primary user-facing macros: `@Dual` (structural) and `@Witness` (DI).

**User-facing surface**: Two primary macros (matching PointFree's two-macro model):
- `@Dual` -- structural duality, both directions, enum infrastructure
- `@Witness` -- DI witness structs with call algebra, observe, unimplemented, mock
- `@Defunctionalize` -- still exists for pure call algebra (academic niche)

**When to apply which**:
- Plain enum: `@Dual`
- Struct with fields, want Dual enum: `@Dual`
- Struct with closures, want DI: `@Witness`
- Struct with closures, want ONLY call algebra: `@Defunctionalize` (rare)

**Advantages**:
1. **Matches PointFree's proven two-macro model.** Users learn two concepts: structural (`@Dual` = `@CasePathable`) and DI (`@Witness` = `@DependencyClient`).
2. **No composition needed.** Each macro is self-contained. Users never stack macros.
3. **Conceptually clean.** `@Dual` = category theory. `@Witness` = software engineering. `@Defunctionalize` = CS theory (available but not primary).
4. **`@Defunctionalize` is honestly scoped.** Its value is the pure reification concept. It is not a building block that other macros compose on -- it is an independent, simpler alternative to `@Witness` for users who do not need DI.

**Disadvantages**:
1. `@Witness` still duplicates `@Defunctionalize`'s Calls enum generation. But this is the same duplication PointFree accepts: `@DependencyClient` generates its own Calls enum without using `@CasePathable`.
2. Three macros still exist, though `@Defunctionalize` is positioned as a secondary/niche tool.
3. Codegen duplication between all three macro implementations remains.

---

### Option G: @Dual is the ONLY Structural Macro, @Witness Only Adds DI

`@Dual` generates enum infrastructure for ANY type (both directions). `@Witness` only adds the DI layer (observe, unimplemented, mock) to witness structs. Internally, `@Witness` uses `@Dual`'s codegen for the Calls enum. `@Defunctionalize` is absorbed into `@Witness` entirely.

This differs from Option F in that `@Defunctionalize` is eliminated as a separate package, and `@Witness` conceptually depends on `@Dual` for its structural component.

**Advantages**:
1. Only two macros. Maximum simplicity.
2. Clean conceptual split: structure (`@Dual`) vs behavior (`@Witness`).

**Disadvantages**:
1. **`@Dual` and `@Defunctionalize` are different operations.** `@Dual` on a struct preserves ALL stored properties including non-function fields (literal type preservation is the duality invariant). `@Defunctionalize`/`@Witness` on a struct extracts ONLY function-typed properties (non-function fields are excluded from the Calls enum). These are genuinely different transformations with different invariants. Merging them conflates structure preservation with behavior extraction.
2. `@Witness` cannot actually use `@Dual`'s codegen for Calls because the selection logic is different (function-only vs all fields, parameter extraction vs literal type).
3. Eliminating `@Defunctionalize` removes a named concept from CS literature (Reynolds 1972) that has independent academic value.

**Verdict**: The structural difference between duality (all fields) and defunctionalization (function fields only) is real and cannot be collapsed. Eliminated.

## Analysis

### The real question has two parts

The analysis reveals two orthogonal decisions:

1. **Conceptual boundaries**: How many user-facing macros, and what does each do?
2. **Implementation sharing**: How is enum infrastructure codegen deduplicated?

These can be decided independently.

### Part 1: Conceptual boundaries

Options C, E, and B are eliminated on technical grounds. Option G is eliminated because duality and defunctionalization have genuinely different invariants. The remaining candidates for conceptual boundaries are:

- **Option A/D**: Three coequal macros (`@Dual`, `@Defunctionalize`, `@Witness`)
- **Option F**: Two primary macros (`@Dual`, `@Witness`) + one niche macro (`@Defunctionalize`)

Option F is strictly better because:

1. **It matches PointFree's proven mental model.** PointFree users learn `@CasePathable` (structural) and `@DependencyClient` (DI). Our users learn `@Dual` (structural) and `@Witness` (DI). `@Defunctionalize` exists for the academically curious but is not a primary tool.

2. **It answers the composition question definitively.** Users never need to stack macros. Each use case maps to exactly one macro:
   - Enum infrastructure: `@Dual`
   - Struct duality (questionnaire pattern): `@Dual`
   - DI witness struct: `@Witness`
   - Pure call algebra (rare): `@Defunctionalize`

3. **`@Witness` subsumption of `@Defunctionalize` is already the reality.** Looking at the implementations, `@Witness`'s Calls enum generation is identical to `@Defunctionalize`'s, plus DI additions. No user who applies `@Defunctionalize` and later needs DI will keep `@Defunctionalize` -- they will switch to `@Witness`. The subsumption is the actual usage pattern.

4. **Three coequal macros creates a false choice.** When a user has a witness struct, they must ask: "Do I use `@Defunctionalize` or `@Witness`?" The answer is always `@Witness` unless they specifically do NOT want DI features. Presenting them as coequal peers overstates `@Defunctionalize`'s utility.

### Part 2: Implementation sharing

Given the conceptual boundary decision (Option F), the implementation sharing question is between:

- **Accept duplication** (as PointFree does): ~280 lines in three places
- **Shared codegen library** (Option D): single source of truth

The `enum-infrastructure-primitives.md` research already analyzed this thoroughly and recommended Option F (shared library at L3). That recommendation stands. However, the **timing** matters:

**Current state**: The codegen files are already duplicated across three packages. All three packages exist and are implemented. The duplication is a maintenance burden TODAY.

**Pragmatic path**: Extract the shared codegen library now. The three consumers already exist, making the case for shared infrastructure clear.

### Answering the key questions

**Q1: Should users ever need to apply more than one of these macros to a single type?**

No. Each use case maps to exactly one macro. There is no composition at the user level.

**Q2: How should @Witness compose with @Dual and/or @Defunctionalize?**

It does not compose with them at the macro level. `@Witness` is self-contained: it generates its own Calls enum (using the same codegen routines as `@Defunctionalize`). `@Dual` is independent. At the implementation level, all three macro packages share codegen via a shared library.

**Q3: What is the minimal set of macros a user needs for each use case?**

| Use Case | Macro | Result |
|----------|-------|--------|
| Witness struct for DI | `@Witness` | Calls, methods, observe, unimplemented, mock |
| Plain enum with infrastructure | `@Dual` | Extraction, Case, Prisms, Scott encoding |
| Call algebra only (no DI) | `@Defunctionalize` | Calls enum with extraction, Case, Prisms |

**Q4: Can Swift macros compose?**

No. Macros cannot see other macros' expansions, cannot apply other macros, and cannot depend on other macros' output at expansion time. Composition must happen at the codegen library level (shared source code) or not at all.

**Q5: What does the dependency graph look like?**


## Outcome

**Status**: RECOMMENDATION

### Recommendation: Option F + Option D

Two orthogonal decisions, both recommended:

**Decision 1 (Conceptual): Option F -- @Witness subsumes @Defunctionalize**

The ecosystem has two primary user-facing macros:

| Macro | Domain | PointFree Analog | Input | Output |
|-------|--------|-----------------|-------|--------|
| `@Dual` | Structural (category theory) | `@CasePathable` | Struct or enum | Dual type + enum infrastructure |
| `@Witness` | DI composition (software engineering) | `@DependencyClient` | Struct with closures | Calls + methods + observe + unimplemented + mock |

`@Defunctionalize` continues to exist as an independent package for the pure call-algebra use case (no DI additions), but it is a niche/academic tool, not a primary user-facing macro. Documentation and onboarding should present `@Dual` and `@Witness` as the two macros users need to know.

Users never stack macros. Each use case maps to exactly one macro.

**Decision 2 (Implementation): Option D -- shared codegen library**

Extract `PrismCodegen.swift`, `CaseDiscriminantCodegen.swift`, `ExtractionCodegen.swift`, and `Utilities.swift` into `swift-enum-infrastructure-codegen` at Layer 3. All three macro implementations import this library. Per the detailed analysis in `enum-infrastructure-primitives.md`.

### Relationship to @Witness enum support removal

Per the recommendation in `witness-enum-direction-and-dual.md`, `@Witness` on enums should be removed. `@Dual` is the correct macro for enums. This is consistent with the present recommendation: `@Witness` is scoped to structs-with-closures (DI witnesses). `@Dual` is scoped to structural duality (both structs and enums).

### Relationship to swift-dependencies

`swift-dependencies` is the DI container. It provides:
- `@Dependency` property wrapper
- `Dependency.Values` (heterogeneous container)
- `Dependency.Context` (mode detection)
- `withDependencies` (scoping)

It depends on `swift-witnesses` (which provides `@Witness`, `Witness.Context`, `Witness.Key`). The relationship is: `@Witness` decorates the types, `swift-dependencies` provides the runtime container.

There is a naming collision to acknowledge: `swift-witnesses` provides `Witness.Key` and `Witness.Context` and `Witness.Values`, while `swift-dependencies` provides `Dependency.Key` and `Dependency.Context` and `Dependency.Values`. These are currently two parallel DI systems. This is a separate concern from the macro composition question and is not addressed here.

### Relationship to swift-dependency-primitives

`swift-dependency-primitives` at L1 provides `Dependency.Key`, `Dependency.Values`, `Dependency.Scope`. These are pure runtime types with no macro dependency. `swift-witness-primitives` at L1 provides `Witness`, `Witness.Protocol`. These are also pure runtime types.

The L1 primitives are correctly scoped: they define the concepts (what a dependency key is, what a witness is). The L3 macros generate code that conforms to these concepts. The L3 DI container provides the runtime infrastructure. Clean layer separation.

### Implementation sequence

1. **Create `swift-enum-infrastructure-codegen`** at `https://github.com/swift-foundations/swift-enum-infrastructure-codegen`. Extract the four shared files. Make all types and functions `public`.

2. **Migrate `swift-dual`** macro implementation to import `Enum_Infrastructure_Codegen`. Remove local copies of `PrismCodegen.swift`, `CaseDiscriminantCodegen.swift`, `ExtractionCodegen.swift`, `Utilities.swift`. Retain `DualMacro.swift`, `StructExpansion.swift`, `EnumExpansion.swift`.

3. **Migrate `swift-defunctionalize`** macro implementation to import `Enum_Infrastructure_Codegen`. Remove local copies. Retain `DefunctionalizeMacro.swift`, `StructExpansion.swift`.
