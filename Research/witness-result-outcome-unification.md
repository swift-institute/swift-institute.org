## Context

The `@Witness` macro generates three DI-specific types as members of the annotated struct:

1. **`Calls: Sendable`** -- one case per closure property, carrying the Copyable arguments. This type is structurally identical to what `@Defunctionalize` would generate (see `dual-defunctionalize-architecture.md`). `Calls` is the shared primitive; `Result` and `Outcome` are the DI-specific additions.

2. **`Result: ~Copyable`** -- one case per closure property, each wrapping `Standard_Library_Extensions.Result<Success, Failure>` with the operation's specific typed-error pair. For example:
   ```swift
   enum Result: ~Copyable {
       case fetch(Standard_Library_Extensions.Result<String, Never>)
       case save(Standard_Library_Extensions.Result<Bool, SaveError>)
       case reset(Standard_Library_Extensions.Result<Void, Never>)
   }
   ```

3. **`Outcome: ~Copyable`** -- pairs `action: Calls` (the call with arguments) with `result: Result` (the typed return value). Exists because after-observers need BOTH what was called and what it returned:
   ```swift
   struct Outcome: ~Copyable {
       let action: Calls
       let result: Result
       consuming func __consumeResult() -> Result { result }
   }
   ```

The observe API uses these types as:
- `before: (Calls) -> Void` -- the call that is about to happen
- `after: (borrowing Outcome) -> Void` -- the call + its result

The generated observe body constructs `Outcome` by capturing the `Calls` value before the real call, then wrapping the return value in `Result`, and finally pairing them in `Outcome`. The after-observer borrows `Outcome` and can consume its result via `__consumeResult()` to pattern-match the typed value.

This research evaluates whether `Result` and `Outcome` should be unified into a single type.

## Question

Should the `Result` and `Outcome` types generated by `@Witness` be merged into a single `Outcome` type where each case carries both the call arguments and the typed result?

## Analysis

### Option A: Keep Both Types (Current)

`Result` and `Outcome` remain separate sibling types of `Calls`.

**Generated shape:**
```swift
struct APIClient {
    // ... closure properties ...

    enum Calls: Sendable { /* one case per operation */ }

    enum Result: ~Copyable {
        case fetch(Standard_Library_Extensions.Result<String, Never>)
        case save(Standard_Library_Extensions.Result<Bool, SaveError>)
    }

    struct Outcome: ~Copyable {
        let action: Calls
        let result: Result
        consuming func __consumeResult() -> Result { result }
    }
}
```

**After-observer consumer pattern:**
```swift
api.observe.after { outcome in
    let action = outcome.action   // what was called
    let result = outcome.__consumeResult()
    switch consume result {
    case .fetch(.success(let value)): /* ... */
    case .save(.failure(let error)): /* ... */
    // ...
    }
}
```

**Advantages:**
- `Result` is independently useful: it can be stored, returned, or forwarded without carrying the arguments.
- `Outcome` is a simple product -- its structure is obvious and its fields are named.
- `__consumeResult()` is a clean consuming extraction. Pattern matching on the `Result` enum gives typed access to success/failure.
- `Result` cases have a 1:1 correspondence with `Calls` cases, making the relationship clear.

**Disadvantages:**
- Three generated types (`Calls`, `Result`, `Outcome`) for every `@Witness` struct. Each adds cognitive overhead and generated code size.
- `Outcome` is a thin wrapper: it holds two fields and a single consuming accessor. It adds indirection without meaningfully abstracting over its components.
- In practice, `Result` is never used independently of `Outcome`. The only consumer of `Result` is `Outcome.__consumeResult()` inside observe bodies. No API path returns a bare `Result`.
- The consumer has to go through two layers to reach the typed value: first extract `Result` from `Outcome`, then pattern-match `Result`.

### Option B: Unified `Outcome` with Arguments + Result Per Case

Merge into a single `Outcome` enum where each case carries both the Copyable arguments and the `Standard_Library_Extensions.Result`:

**Generated shape:**
```swift
struct APIClient {
    enum Calls: Sendable { /* unchanged */ }

    enum Outcome: ~Copyable {
        case fetch(id: Int, Standard_Library_Extensions.Result<String, Never>)
        case save(data: String, Standard_Library_Extensions.Result<Bool, SaveError>)
        case reset(Standard_Library_Extensions.Result<Void, Never>)
    }
}
```

**After-observer consumer pattern:**
```swift
api.observe.after { outcome in
    switch consume outcome {
    case .fetch(let id, .success(let value)):
        print("fetch(\(id)) returned \(value)")
    case .save(_, .failure(let error)):
        print("save failed: \(error)")
    // ...
    }
}
```

**Advantages:**
- Two generated types instead of three. Simpler mental model.
- Direct pattern matching: one `switch` gives you both the arguments and the result. No intermediate extraction step.
- The `Outcome` type is self-contained -- each case tells the complete story of what happened.
- No `__consumeResult()` consuming accessor needed. The `borrowing Outcome` in the observer signature would need to become `consuming Outcome` for the pattern match, but the ownership transfer is explicit in the generated closure body.
- Generated code shrinks: no `Result` enum, no `Outcome` struct, no `__consumeResult()`. Just one enum.

**Disadvantages:**
- Duplicates argument information between `Calls` and `Outcome`. The `id: Int` in `.fetch(id:)` appears in both types. However, these are Copyable values and the duplication is structural (codegen), not runtime (the arguments are copied once when constructing `Outcome`).
- Loses the ability to handle a result independently of its arguments. In practice this is not a current use case -- `Result` is always accessed through `Outcome`.
- The `after` observer callback signature must change. Currently `after: (borrowing Outcome) -> Void` borrows the struct and the consumer calls `__consumeResult()`. With a unified enum, the consumer needs to `switch consume outcome`. The callback type becomes `after: (consuming Outcome) -> Void`. This is a different ownership convention, though both are generated code and not user-facing signatures.
- Cases with many parameters become wide: `case poll(handle: borrowing NoncopyableHandle, buffer: inout [UInt8], Standard_Library_Extensions.Result<Int, PollError>)`. However, owned parameters are already omitted (the omission pattern from `witness-macro-noncopyable-support-design.md`), so only Copyable parameters appear, mitigating width.

### Option C: Unified `Outcome` with Nested `Result`

Keep `Result` as a nested type inside `Outcome`, but eliminate the standalone `Result` sibling:

**Generated shape:**
```swift
struct APIClient {
    enum Calls: Sendable { /* unchanged */ }

    enum Outcome: ~Copyable {
        case fetch(id: Int, Result)
        case save(data: String, Result)
        case reset(Result)

        enum Result: ~Copyable {
            case fetch(Standard_Library_Extensions.Result<String, Never>)
            case save(Standard_Library_Extensions.Result<Bool, SaveError>)
            case reset(Standard_Library_Extensions.Result<Void, Never>)
        }
    }
}
```

**After-observer consumer pattern:**
```swift
api.observe.after { outcome in
    switch consume outcome {
    case .fetch(let id, .fetch(.success(let value))):
        print("fetch(\(id)) returned \(value)")
    // ...
    }
}
```

**Advantages:**
- `Result` is scoped under `Outcome`, signaling its subordinate role.
- Still only two top-level types.

**Disadvantages:**
- Deeply nested pattern matching: `case .fetch(let id, .fetch(.success(let value)))` -- the `.fetch` appears twice because the `Outcome` case and the `Result` case share the same name for the same operation. This is significantly worse ergonomics than either Option A or B.
- The consumer can mis-pair: `case .fetch(_, .save(.success(...)))` compiles but is semantically nonsensical. With Option B's flat structure, this is impossible because arguments and result are in the same case.
- `Result` is still generated as a full enum, so generated code volume is similar to Option A.
- The nesting creates a structural guarantee problem: nothing enforces that `Outcome.fetch` always contains `Result.fetch`. The type system allows mismatched pairs.

### Comparison

| Criterion | A: Separate Types | B: Unified Outcome | C: Nested Result |
|-----------|-------------------|-------------------|-----------------|
| Generated type count | 3 (`Calls`, `Result`, `Outcome`) | 2 (`Calls`, `Outcome`) | 2 (`Calls`, `Outcome` with nested `Result`) |
| Pattern match depth | 2 steps (extract Result, then match) | 1 step (direct match) | 2 steps (match Outcome case, then match nested Result) |
| Pattern match ergonomics | Moderate | **Best** | Worst (double case name) |
| Mispairing risk | None (struct fields) | None (single case) | Yes (Outcome.fetch can hold Result.save) |
| Result independence | Possible (unused in practice) | Not possible | Possible (unused in practice) |
| Argument duplication | None | Structural (codegen only) | Structural (codegen only) |
| Generated code size | Largest (enum + struct + accessor) | **Smallest** (single enum) | Medium (enum + nested enum) |
| ~Copyable handling | `consuming func __consumeResult()` | `consuming` pattern match | `consuming` pattern match + nested consuming |
| Observer signature | `borrowing Outcome` | `consuming Outcome` | `consuming Outcome` |
| Breaking change to macro | None | Moderate (remove Result, restructure Outcome, update observe body codegen) | Moderate (nest Result, restructure Outcome, update observe body codegen) |
| Mental model | Three types, clear roles | Two types, self-contained | Two types, redundant structure |

### Observer Signature Impact

The shift from `borrowing Outcome` (Option A) to `consuming Outcome` (Option B) deserves examination.

In Option A, the after-observer borrows `Outcome` and calls `__consumeResult()` which consumes it. This two-step dance exists because the observer *borrows* the struct but needs to *consume* the `Result` field. The generated observe body creates the `Outcome`, passes it to the callback (borrow), then the callback extracts the result.

In Option B, the after-observer consumes `Outcome` directly via pattern matching. The generated observe body creates the `Outcome` enum value and passes ownership to the callback. This is simpler: the callback consumes the value and matches it in one operation.

Both approaches are internal to the generated code. The user writes `api.observe.after { outcome in ... }` regardless. The ownership convention is determined by the macro and invisible at the call site. The change is therefore non-breaking for user-authored code.

### Generated Observe Body Comparison

**Option A (current) -- non-throwing, non-void:**
```swift
// Generated by macro inside observe closure:
let action: Calls = .fetch(id: id)
before(action)
let result = witness._fetch(id)
let __outcome = Outcome(
    action: action,
    result: Result.fetch(Standard_Library_Extensions.Result.success(result))
)
after(__outcome)
let __result = __outcome.__consumeResult()
switch consume __result {
case .fetch(.success(let __value)): return __value
default: fatalError("unreachable")
}
```

**Option B (unified) -- non-throwing, non-void:**
```swift
// Generated by macro inside observe closure:
let action: Calls = .fetch(id: id)
before(action)
let result = witness._fetch(id)
let __outcome = Outcome.fetch(
    id: id,
    Standard_Library_Extensions.Result.success(result)
)
after(__outcome)
// after consumed __outcome; 'result' is still live from the let binding
return result
```

Option B is notably simpler in the generated body. Because `result` is captured in a local before being wrapped in `Outcome`, the generated code can return `result` directly after the observer consumes `Outcome`. There is no need to re-extract the value from the consumed Outcome -- the macro generated the local binding and owns it.

This eliminates:
1. The `__consumeResult()` call
2. The `switch consume __result` with `fatalError("unreachable")`
3. The `Result` enum entirely

### Before-Observer Compatibility

The `before` observer takes `(Calls) -> Void` and is unaffected by this change. `Calls` remains the same type regardless of which option is chosen. The question only concerns `Result` and `Outcome`.

## Outcome

**Status**: RECOMMENDATION

**Recommendation**: Adopt Option B -- Unified `Outcome` with arguments + result per case.

**Rationale**:

1. **`Result` has no independent consumers.** In the current codebase, `Result` is only ever constructed inside the generated observe body and immediately placed into an `Outcome`. No API path returns a standalone `Result`. The type exists solely as a component of `Outcome`, which means it has no independent justification.

2. **Pattern matching ergonomics improve significantly.** Option B reduces the consumer from a two-step extract-then-match to a single `switch consume outcome`. The unified case `case .fetch(let id, .success(let value))` reads as a natural sentence: "the fetch of id succeeded with value."

3. **Generated code becomes simpler.** The macro no longer generates a `Result` enum, an `Outcome` struct, or a `__consumeResult()` accessor. Instead it generates one enum with one case per operation. The generated observe body avoids the re-extraction dance (`__consumeResult()` followed by `switch` with `fatalError("unreachable")`).

4. **The argument duplication is benign.** The Copyable arguments appear in both `Calls` and `Outcome`, but this is structural duplication in generated code. At runtime, the arguments are copied once when constructing `Outcome`. Given that only Copyable parameters appear (per the omission pattern), this is always a value copy of simple types.

5. **Option C is strictly worse.** Nesting `Result` inside `Outcome` creates double-named pattern matching (`.fetch(_, .fetch(.success(...)))`) and introduces a mispairing risk where the Outcome case and Result case can diverge. Option B avoids both problems.

6. **The breaking change is internal.** No user-authored code references `Result` or `Outcome` directly -- they appear only in generated macro output. Changing the macro's output structure affects the macro implementation (`WitnessMacro.swift`), not consumer code.

**Implementation path**:

1. Remove `generateResultEnum()` and `generateTypedResultCase()` from `WitnessMacro.swift` (lines 1020-1038).
2. Replace `Outcome` struct generation (lines 927-942 in `generateCallsMembers`) with an `Outcome` enum generation function that produces one case per closure property carrying both the Copyable arguments and the `Standard_Library_Extensions.Result<Success, Failure>`.
3. Update `generateObserveBody()` (lines 1292-1390) to construct `Outcome` enum cases directly and return the locally-bound result instead of re-extracting from the consumed Outcome.
4. Change the after-observer parameter from `borrowing Outcome` to `consuming Outcome` in `generateObserveStruct()`.
5. Update the test fixtures and observe tests to use the new pattern matching shape.
