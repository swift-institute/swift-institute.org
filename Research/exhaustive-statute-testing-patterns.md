## Context

The `swift-nl-wetgever` statute packages encode Dutch legal articles as Swift types. Each article's `Lid` (subsection) has an `Arguments` struct where every stored property is `Bool?` ŌĆö representing three-valued legal conditions (`true`, `false`, `nil`/unknown).

Testing requires exhaustive verification: every combination of `Bool?` inputs must produce the correct legal conclusion. For a `Lid` with N conditions, this means 3^N test cases.

The current pattern relies on `swift-test-primitives` infrastructure that extends `Array` of `Bool?` tuples with `.allCases` static properties (2-tuples through 6-tuples), producing all combinations.

**Trigger**: During test restructuring of `burgerlijk-wetboek-boek-2`, the pattern's verbosity and type-unsafety became apparent.

## Question

Can we improve the exhaustive testing pattern to:
1. Eliminate positional tuple access (`.0`, `.1`, `.2`, ...)?
2. Use the type's own structure instead of parallel tuple types?
3. Reduce manual synchronization between argument count and tuple arity?

## Analysis

### Current Pattern (Status Quo)

```swift
// Test Primitives infrastructure (BoolOptional+CaseIterable.swift)
extension [(Bool?, Bool?, Bool?, Bool?, Bool?)] {
    public static let allCases: Self = [(Bool?, Bool?, Bool?, Bool?)].allCases.flatMap {
        first, second, third, fourth in
        Bool?.allCases.map { fifth in (first, second, third, fourth, fifth) }
    }
}

// Test call site
@Test(arguments: [(Bool?, Bool?, Bool?, Bool?, Bool?)].allCases)
func `Lid 1 bepaalt rechtspersoonlijkheid`(argument: (Bool?, Bool?, Bool?, Bool?, Bool?)) throws {
    let result = try `Artikel 1`.`Lid 1`(
        `betreft het de staat`: argument.0,
        `betreft het een provincie`: argument.1,
        `betreft het een gemeente`: argument.2,
        `betreft het een waterschap`: argument.3,
        `is het een lichaam waaraan krachtens de Grondwet verordenende bevoegdheid is verleend`: argument.4
    )

    switch argument {
    case (true, _, _, _, _), (_, true, _, _, _), (_, _, true, _, _),
         (_, _, _, true, _), (_, _, _, _, true):
        #expect(true == result.`het bezit rechtspersoonlijkheid`)
    case (_, _, _, _, _):
        #expect(nil == result.`het bezit rechtspersoonlijkheid`)
    }
}
```

**Advantages**:
- Works today, no additional infrastructure needed
- Tuple pattern matching in `switch` is concise
- 243 test cases generated correctly (3^5)

**Disadvantages**:
- Positional access (`.0`ŌĆō`.4`) carries no meaning ŌĆö easy to swap arguments silently
- Tuple arity must match argument count manually ŌĆö adding a 6th condition to the type requires changing the tuple type everywhere
- Assertion logic uses positional patterns, not named conditions
- The tuple type `(Bool?, Bool?, Bool?, Bool?, Bool?)` duplicates information already in `Arguments`

### Option A: `Arguments: CaseIterable`

Make the `Arguments` struct conform to `CaseIterable`, producing all 3^N combinations of its `Bool?` properties.

```swift
// In Test Support (or generated by @Splat)
extension `Artikel 1`.`Lid 1`.Arguments: CaseIterable {
    public static var allCases: [Self] {
        Bool?.allCases.flatMap { a in
            Bool?.allCases.flatMap { b in
                Bool?.allCases.flatMap { c in
                    Bool?.allCases.flatMap { d in
                        Bool?.allCases.map { e in
                            Self(
                                `betreft het de staat`: a,
                                `betreft het een provincie`: b,
                                `betreft het een gemeente`: c,
                                `betreft het een waterschap`: d,
                                `is het een lichaam waaraan krachtens de Grondwet verordenende bevoegdheid is verleend`: e
                            )
                        }
                    }
                }
            }
        }
    }
}
```

**Test call site**:
```swift
@Test(arguments: `Artikel 1`.`Lid 1`.Arguments.allCases)
func `Lid 1 bepaalt rechtspersoonlijkheid`(arguments: `Artikel 1`.`Lid 1`.Arguments) throws {
    let result = try `Artikel 1`.`Lid 1`(arguments)

    if arguments.`betreft het de staat` == true
        || arguments.`betreft het een provincie` == true
        || arguments.`betreft het een gemeente` == true
        || arguments.`betreft het een waterschap` == true
        || arguments.`is het een lichaam waaraan krachtens de Grondwet verordenende bevoegdheid is verleend` == true
    {
        #expect(true == result.`het bezit rechtspersoonlijkheid`)
    } else {
        #expect(nil == result.`het bezit rechtspersoonlijkheid`)
    }
}
```

**Advantages**:
- Named property access ŌĆö no positional ambiguity
- `arguments` passes directly to `init(_: Arguments)` ŌĆö no manual field mapping
- Adding a property to `Arguments` automatically extends `allCases` (when macro-generated)
- Assertion reads like the statute: "als het de staat betreft, of een provincie, ..."
- Type-safe: the compiler enforces that the test operates on the correct `Arguments` type

**Disadvantages**:
- Verbose manual conformance (N levels of nested `flatMap`)
- Assertion logic mirrors implementation ŌĆö but this is inherent to exhaustive testing; the switch-on-tuples pattern has the same problem, just with positional encoding
- Loses tuple pattern matching syntax (must use `if`/`switch` on named properties)

### Option B: `@Splat` Generates `CaseIterable` on `Arguments`

Extend the `@Splat` macro to generate `CaseIterable` conformance on `Arguments` when all stored properties conform to `CaseIterable`.

```swift
// Source code ŌĆö no change needed
@Splat
public struct `Lid 1`: Sendable {
    public let arguments: Arguments
    public let `het bezit rechtspersoonlijkheid`: Bool?

    public struct Arguments: Sendable {
        public let `betreft het de staat`: Bool?
        // ... (all Bool?)
    }
    // ...
}

// @Splat generates (in addition to the splatted init):
extension `Artikel 1`.`Lid 1`.Arguments: CaseIterable {
    public static var allCases: [Self] { /* flatMap chain */ }
}
```

**Test call site**: Same as Option A, but with zero manual conformance code.

**Advantages**:
- Zero additional code per type ŌĆö @Splat already knows all Arguments properties
- Automatically stays in sync when properties change
- CaseIterable conformance is a natural extension of the @Splat pattern

**Disadvantages**:
- Puts test-only functionality in the production `@Splat` macro
- CaseIterable conformance on production types may be misleading ŌĆö it's primarily a test affordance
- Requires `@Splat` to detect whether all properties conform to `CaseIterable` at macro expansion time
- @Splat is in `swift-splat`, a general-purpose package ŌĆö statute-specific behavior may not belong there

**Mitigation**: Gate behind a parameter: `@Splat(caseIterable: true)` or `@Splat(exhaustive: true)`.

### Option C: Dedicated `@ExhaustiveCases` Macro on Arguments

A separate, lightweight macro that generates `CaseIterable` for any struct whose stored properties all conform to `CaseIterable`.

```swift
@Splat
public struct `Lid 1`: Sendable {
    public let arguments: Arguments

    @ExhaustiveCases
    public struct Arguments: Sendable {
        public let `betreft het de staat`: Bool?
        public let `betreft het een provincie`: Bool?
        // ...
    }
    // ...
}
```

**Advantages**:
- Separation of concerns ŌĆö @Splat does splatting, @ExhaustiveCases does case generation
- Opt-in per type ŌĆö not every Arguments needs it
- Could live in `swift-test-primitives` alongside the existing Bool? infrastructure
- Generalizes beyond statute testing ŌĆö any struct with CaseIterable properties

**Disadvantages**:
- Additional macro dependency
- Two macros on Arguments might feel heavy
- Must be applied manually to each Arguments type

### Option D: Test Support Extension (Manual, No Macro)

Add CaseIterable conformances manually in each statute package's Test Support module.

```swift
// Tests/Support/exports.swift
@_exported public import Burgerlijk_Wetboek_Boek_2
@_exported public import Test_Primitives

// Tests/Support/Burgerlijk Wetboek Boek 2 Test Support.swift
extension `Artikel 1`.`Lid 1`.Arguments: CaseIterable {
    public static var allCases: [Self] { /* ... */ }
}
extension `Artikel 1`.`Lid 2`.Arguments: CaseIterable {
    public static var allCases: [Self] { /* ... */ }
}
// ... per Lid
```

**Advantages**:
- No macro changes needed ŌĆö works today
- Test-only code stays in Test Support
- Follows existing [TEST-019] pattern

**Disadvantages**:
- Verbose boilerplate per type (N-level nested flatMap for each Lid)
- Must be updated manually when Arguments change
- Lots of repetitive code across statute packages

## Comparison

| Criterion | Status Quo | A: Manual CaseIterable | B: @Splat generates | C: @ExhaustiveCases | D: Test Support |
|-----------|-----------|----------------------|--------------------|--------------------|----------------|
| Named access | No (`.0`, `.1`) | Yes | Yes | Yes | Yes |
| Type safety | Positional | Named | Named | Named | Named |
| Sync with type | Manual | Manual | Automatic | Automatic | Manual |
| Code per Lid | ~0 (uses tuples) | ~15 lines | 0 | ~1 line annotation | ~15 lines |
| New infrastructure | None | None | @Splat change | New macro | None |
| Test-only concern | Yes (in test-primitives) | Yes (production type) | Configurable | Yes (test macro) | Yes (Test Support) |
| Generalizes | No (tuple-arity bound) | Yes | Yes (statute types) | Yes (any struct) | No (manual) |

## Constraints

1. **@Test(arguments:) requires `Collection & Sendable`** with `Sendable` elements. `[Arguments]` satisfies this since Arguments is Sendable.
2. **Maximum 2 argument collections** in Swift Testing. A single `Arguments.allCases` collection works within this limit.
3. **Swift Testing limitation**: `@Suite` in extensions of generic type specializations is silently not discovered ([swiftlang/swift-testing#1508](https://github.com/swiftlang/swift-testing/issues/1508)). Not relevant here ŌĆö `Arguments` is non-generic.
4. **Bool? is already CaseIterable** via `swift-test-primitives` (`@retroactive CaseIterable`). Arguments.allCases can build on this.
5. **@Splat macro expansion** happens at compile time and has access to all stored property declarations. Generating `CaseIterable` is mechanically feasible.

## Outcome

**Status**: RECOMMENDATION

**Short-term (implement now)**: **Option D** ŌĆö Manual `CaseIterable` conformances in Test Support. This requires no new infrastructure and follows existing conventions. The boilerplate is mechanical and can be done once per statute package.

**Medium-term (next @Splat release)**: **Option C** ŌĆö A dedicated `@ExhaustiveCases` macro in `swift-test-primitives`. This generalizes the pattern, keeps test concerns separate from production code, and eliminates manual synchronization. The macro is straightforward: inspect stored properties, verify all conform to `CaseIterable`, emit nested `flatMap`/`map` chain.

**Deferred**: Option B (@Splat integration). Revisit if @ExhaustiveCases proves insufficient or if the pattern becomes universal enough to justify coupling with @Splat.

**Not recommended**: Option A (manual CaseIterable without Test Support placement). Putting test conformances on production types without clear separation violates the Test Support architecture.

### Implementation Notes for Option D (Short-term)

The test call site transforms from:

```swift
// Before
@Test(arguments: [(Bool?, Bool?, Bool?, Bool?, Bool?)].allCases)
func `Lid 1 bepaalt rechtspersoonlijkheid`(argument: (Bool?, Bool?, Bool?, Bool?, Bool?)) throws {
    let result = try `Artikel 1`.`Lid 1`(
        `betreft het de staat`: argument.0,
        `betreft het een provincie`: argument.1,
        `betreft het een gemeente`: argument.2,
        `betreft het een waterschap`: argument.3,
        `is het een lichaam waaraan krachtens de Grondwet verordenende bevoegdheid is verleend`: argument.4
    )
    // ...
}

// After
@Test(arguments: `Artikel 1`.`Lid 1`.Arguments.allCases)
func `Lid 1 bepaalt rechtspersoonlijkheid`(arguments: `Artikel 1`.`Lid 1`.Arguments) throws {
    let result = try `Artikel 1`.`Lid 1`(arguments)
    // ...
}
```

Key improvements at the call site:
- **3 lines** replace **7 lines** (argument ŌåÆ init mapping eliminated)
- Named properties in assertions instead of positional tuple access
- Direct `Arguments` ŌåÆ `init` passthrough

### Implementation Notes for Option C (Medium-term)

The `@ExhaustiveCases` macro would:
1. Inspect all stored properties of the annotated struct
2. Verify each property's type conforms to `CaseIterable`
3. Emit `CaseIterable` conformance with nested `flatMap`/`map` chain
4. Compile-time error if any property is not `CaseIterable`

```swift
// Macro usage
@ExhaustiveCases
public struct Arguments: Sendable {
    public let `betreft het de staat`: Bool?
    public let `betreft het een provincie`: Bool?
}

// Generated
extension Arguments: CaseIterable {
    public static var allCases: [Self] {
        Bool?.allCases.flatMap { v0 in
            Bool?.allCases.map { v1 in
                Self(`betreft het de staat`: v0, `betreft het een provincie`: v1)
            }
        }
    }
}
```
