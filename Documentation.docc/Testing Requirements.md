Test organization patterns using `#TestSuites` and performance testing.

## Overview

This document defines the project-wide requirements for test organization using the `#TestSuites` macro. These rules apply across all packages and targets unless an explicit, reviewed exception is recorded.

### Testing Framework Status

**Current:** Use `Testing Extras` from [swift-testing-extras](https://github.com/coenttb/swift-testing-extras). This package extends Apple's Swift Testing framework with the `#TestSuites` macro, performance traits, and exclusion utilities.

**Future:** The `Testing` module from [swift-testing](https://github.com/coenttb/swift-testing) is under active development. Once finalized, it will become the primary testing framework, providing a `#Tests` macro with additional suite categories (Snapshot) and deeper integration with the standards infrastructure.

```swift
// Current (swift-testing-extras)
import Testing Extras

// Future (swift-testing) - not yet finalized
import Testing
```

Both frameworks follow the same organizational patterns documented below.

**Normative language**: This document uses RFC 2119 conventions:
- **MUST** / **MUST NOT**: Absolute requirement or prohibition
- **SHOULD** / **SHOULD NOT**: Recommended unless valid reason exists
- **MAY**: Optional

## Test Suite Organization

**Applies to**: All test files in all packages.

**Does not apply to**: Benchmark-only targets or third-party test utilities.

---

### [TEST-ORG-001] TestSuites Macro Declaration

**Scope**: All types requiring tests.

**Statement**: Every type being tested MUST declare `#TestSuites` in an extension within the test target.

**Correct** (current):
```swift
import Testing Extras

@testable import YourModule

extension YourType {
    #TestSuites
}
```

**Correct** (future, once swift-testing is finalized):
```swift
import Testing

@testable import YourModule

extension YourType {
    #Tests
}
```

**Incorrect**:
```swift
// Ad-hoc test struct without #TestSuites
@Suite struct YourTypeTests {
    @Test func someTest() { ... }
}
```

The macro generates a nested test structure.

**Current** (`#TestSuites` from Testing Extras):
```swift
extension YourType {
    @Suite enum Test {
        @Suite(.exclusive(group: "YourModule.YourType")) struct Unit {}
        @Suite(.exclusive(group: "YourModule.YourType")) struct EdgeCase {}
        @Suite(.exclusive(group: "YourModule.YourType")) struct Integration {}
        @Suite(.exclusive, .serialized) struct Performance {}
    }
}
```

**Future** (`#Tests` from swift-testing):
```swift
extension YourType {
    @Suite enum Test {
        @Suite struct Unit {}
        @Suite struct EdgeCase {}
        @Suite struct Integration {}
        @Suite(.serialized) struct Performance {}
        @Suite struct Snapshot {}
    }
}
```

**Rationale**: Consistent test structure enables predictable navigation and ensures all types have Unit, EdgeCase, Integration, and Performance test categories.

**Cross-references**: [TEST-ORG-002], [API-NAME-001]

---

### [TEST-ORG-002] Generated Suite Structure

**Scope**: The `Test` enum generated by `#TestSuites` or `#Tests`.

**Statement**: Tests MUST be added by extending the generated nested types. Custom suites MAY be added to `Type.Test` for domain-specific categories.

**Generated Suite Categories**:

| Suite | Purpose |
|-------|---------|
| `Unit` | Standard functionality tests |
| `EdgeCase` | Boundary conditions, error paths |
| `Integration` | Cross-type interactions |
| `Performance` | Timing and allocation tracking |
| `Snapshot` | Snapshot comparison tests (future, swift-testing only) |

**Correct**:
```swift
extension YourType.Test.Unit {
    @Test("description of what is tested")
    func testName() throws {
        // test implementation
    }
}

extension YourType.Test.EdgeCase {
    @Test("edge case description")
    func edgeCaseName() throws {
        // edge case test
    }
}

extension YourType.Test.Integration {
    @Test("integration scenario description")
    func integrationTest() throws {
        // integration test
    }
}

extension YourType.Test.Performance {
    @Test("performance scenario", .timed(iterations: 100, warmup: 10))
    func performanceTest() {
        // performance test
    }
}
```

**Incorrect**:
```swift
// Tests outside the generated structure
@Test func testYourType() { ... }
```

**Rationale**: Extending generated types maintains hierarchical organization and enables selective test execution by category.

**Cross-references**: [TEST-ORG-001], [TEST-ORG-003]

---

### [TEST-ORG-003] Additional Test Suites

**Scope**: Custom test categories beyond Unit, EdgeCase, and Performance.

**Statement**: Additional suites SHOULD be added as extensions to `Type.Test` when domain-specific categories are needed. Traits such as `@MainActor` or `.serialized` MUST be applied as required by the test context.

**Correct**:
```swift
extension YourType.Test {
    @MainActor
    @Suite(.serialized) struct Snapshot {}
}

extension YourType.Test {
    @Suite(.serialized) struct `Foundation Comparison` {}
}
```

**Rationale**: Extensible suite structure accommodates specialized testing needs while maintaining organizational consistency.

**Cross-references**: [TEST-ORG-002]

---

### [TEST-ORG-004] Nested Type Testing

**Scope**: Types nested within other types (e.g., `File.Path.Component`).

**Statement**: Nested types MUST declare `#TestSuites` independently. The test hierarchy MUST mirror the type hierarchy.

**Correct**:
```swift
// Type hierarchy: File.Path.Component
extension File.Path.Component {
    #TestSuites
}

extension File.Path.Component.Test.Unit {
    @Test("component parsing")
    func componentParsing() throws { ... }
}
```

This creates the test hierarchy:
- `File.Path.Component.Test.Unit`
- `File.Path.Component.Test.EdgeCase`
- `File.Path.Component.Test.Integration`
- `File.Path.Component.Test.Performance`

**Rationale**: Mirroring type hierarchy in tests enables predictable test location and supports the one-type-per-file principle.

**Cross-references**: [TEST-FILE-001], [API-IMPL-005]

---

### [TEST-ORG-005] Protocol Testing

**Scope**: Testing protocol types.

**Statement**: Protocols MUST NOT use `#TestSuites` directly. Protocol tests MUST be placed in the parent type's test file under the appropriate suite category.

**Rationale**: Swift protocols cannot have nested types added via extensions, making `extension SomeProtocol { #TestSuites }` impossible. Protocol tests belong with their containing type.

**Correct**:
```swift
// Source: Witness.swift contains Witness and Witness.Protocol
// Test file: Witness Tests.swift

extension Witness {
    #TestSuites
}

// Protocol conformance tests go in the parent's Unit suite
extension Witness.Test.Unit {
    @Test("Protocol requirement satisfaction")
    func protocolRequirements() throws {
        // Test that concrete types satisfy Witness.Protocol
    }

    @Test("Protocol default implementation")
    func protocolDefaults() throws {
        // Test protocol extension default behavior
    }
}
```

**Incorrect**:
```swift
// Cannot add nested types to protocols
extension Witness.Protocol {
    #TestSuites  // Compile error: protocols cannot have nested types
}
```

**Guidelines**:

| Protocol Location | Test Location |
|-------------------|---------------|
| `Witness.Protocol` | `Witness Tests.swift` → `Witness.Test.Unit` |
| `File.Handle.Protocol` | `File.Handle Tests.swift` → `File.Handle.Test.Unit` |
| Top-level `SomeProtocol` | Create a companion test enum or test in consuming types |

For top-level protocols without a parent type, tests MAY be organized by:
1. Testing through concrete conforming types
2. Creating a dedicated test enum: `enum SomeProtocolTests { #TestSuites }`

**Cross-references**: [TEST-ORG-004], [TEST-FILE-001]

## Test File Organization

**Applies to**: All test file naming and placement.

**Does not apply to**: Generated test files or external test frameworks.

---

### [TEST-FILE-001] File Naming Convention

**Scope**: All test files in `/Tests/` directories.

**Statement**: Test files MUST be named `Type.Path Tests.swift`, mirroring the type hierarchy. Each file in `/Sources/` MUST have a corresponding file in `/Tests/`.

**Correct**:
```
Tests/
  Your Module Tests/
    File Tests.swift                      -> File
    File.Handle Tests.swift               -> File.Handle
    File.Path Tests.swift                 -> File.Path
    File.Path.Component Tests.swift       -> File.Path.Component
    File.System.Metadata.Info Tests.swift -> File.System.Metadata.Info
```

**Incorrect**:
```
Tests/
  Your Module Tests/
    FileTests.swift           // Missing space before "Tests"
    PathTests.swift           // Missing type hierarchy
    AllFileTests.swift        // Combined tests
```

**Rationale**: Predictable file naming enables immediate navigation between source and test files.

**Cross-references**: [TEST-ORG-004], [API-IMPL-005]

---

### [TEST-FILE-002] Test Target Mirroring

**Scope**: Test target organization in `Package.swift`.

**Statement**: Test targets MUST mirror source targets with the suffix " Tests". One test target per source target.

**Correct**:
```
Sources/                    Tests/
  File System               File System Tests
  File System Primitives    File System Primitives Tests
  File System Async         File System Async Tests
```

**Incorrect**:
```
Sources/                    Tests/
  File System               FileSystemTests        // Wrong naming
  File System Primitives    (no test target)       // Missing tests
```

**Rationale**: Parallel target structure ensures complete test coverage and simplifies dependency management.

**Cross-references**: [API-LAYER-001]

---

### [TEST-FILE-003] Themed Integration Tests

**Scope**: Integration scenarios spanning multiple types.

**Statement**: Themed tests MAY be added for integration scenarios. Themed tests SHOULD use `#TestSuites` when centered on a specific type or concept.

**Correct**:
```swift
// For integration scenarios centered on a concept
extension Integration {
    #TestSuites
}

extension Integration.Test.Unit {
    @Test("end-to-end file operation")
    func endToEndFileOperation() throws { ... }
}
```

**Rationale**: Integration tests verify cross-type interactions while maintaining organizational consistency.

**Cross-references**: [TEST-ORG-001]

## Performance Testing

**Applies to**: All performance tests using Testing Extras (current) or Testing (future).

**Does not apply to**: External benchmarking tools or profiling sessions.

---

### [TEST-PERF-001] Performance Test Location

**Scope**: All performance tests.

**Statement**: Performance tests MUST live in `Type.Test.Performance` (generated by `#TestSuites` or `#Tests`). Performance APIs are provided by `Testing Extras` (current) or `Testing` (future).

**Correct**:
```swift
extension File.Path.Test.Performance {
    @Test("path appending", .timed(iterations: 100, warmup: 10))
    func pathAppending() {
        var path = File.Path.root
        for i in 0..<100 {
            path = path.appending("dir\(i)")
        }
    }
}
```

**Incorrect**:
```swift
// Performance test outside Performance suite
extension File.Path.Test.Unit {
    @Test("path appending", .timed(iterations: 100))
    func pathAppending() { ... }
}
```

**Rationale**: Centralized performance tests enable selective execution and consistent measurement configuration.

**Cross-references**: [TEST-ORG-002], [API-TEST-001]

---

### [TEST-PERF-002] Timed Test Configuration

**Scope**: The `.timed()` trait parameters.

**Statement**: Timed tests MUST specify `iterations`. Warmup runs SHOULD be specified for stable measurements. Thresholds SHOULD be set for regression detection.

**Parameters** (Testing Extras):

| Parameter | Type | Default | Purpose |
|-----------|------|---------|---------|
| `iterations` | `Int` | 10 | Number of measured runs |
| `warmup` | `Int` | 0 | Untimed warmup runs |
| `threshold` | `Duration?` | nil | Budget (fails if exceeded) |
| `metric` | `Metric` | `.median` | Which metric to check |

> **Note**: Memory tracking parameters (`maxAllocations`, `detectLeaks`, `peakMemoryLimit`) are planned for future swift-testing releases.

**Correct**:
```swift
@Test("must complete within 50ms", .timed(iterations: 50, threshold: .milliseconds(50)))
func fastOperation() {
    // Fails if median exceeds 50ms
}
```

**Important**: Swift Testing creates a new test struct instance for each iteration. If you need stable state across iterations, use `Benchmark.measure()` instead:

```swift
@Test("stable state benchmark")
func stableStateBenchmark() throws {
    let fixture = setupExpensiveFixture()
    let measurement = try Benchmark.measure(iterations: 10, warmup: 2) {
        fixture.performOperation()
    }
    // measurement contains statistics: min, max, median, mean, percentiles
}
```

**Cross-references**: [TEST-PERF-003], [TEST-PERF-004]

### [TEST-PERF-003] Metric Selection

**Scope**: Choosing the appropriate metric for threshold enforcement.

**Statement**: Tests SHOULD use `.median` (default) for general performance. Tests MUST use `.p99` or `.p95` for latency-sensitive operations.

**Available Metrics**:

| Metric | Purpose |
|--------|---------|
| `.min` | Fastest iteration |
| `.max` | Slowest iteration |
| `.median` | 50th percentile (default, most stable) |
| `.mean` | Average duration |
| `.p95` | 95th percentile |
| `.p99` | 99th percentile (tail latency) |

**Correct**:
```swift
@Test("p99 SLO check", .timed(iterations: 100, threshold: .milliseconds(100), metric: .p99))
func latencySensitive() {
    // Fails if p99 exceeds 100ms
}
```

**Rationale**: Metric selection determines what behavior is enforced. Median is stable; percentiles catch outliers.

**Cross-references**: [TEST-PERF-002]

---

### [TEST-PERF-004] Memory and Allocation Testing

**Scope**: Tests verifying memory behavior.

**Statement**: Allocation-sensitive code SHOULD specify memory constraints when APIs become available.

> **Note**: Memory tracking APIs (`maxAllocations`, `detectLeaks`, `peakMemoryLimit`) are planned for future swift-testing releases. The examples below show the intended API design.

**Planned API** (future swift-testing):
```swift
// Allocation tracking
@Test("low allocation operation", .timed(iterations: 10, maxAllocations: 1024))
func minimalAllocations() {
    // Fails if median allocation exceeds 1KB
}

// Memory leak detection
@Test("no leaks", .timed(), .detectLeaks())
func noMemoryLeaks() {
    // Fails if net allocations remain after test
}

// Peak memory tracking
@Test("bounded memory", .timed(), .trackPeakMemory(limit: 10_000_000))
func stayUnderMemoryBudget() {
    // Fails if peak memory exceeds 10MB
}
```

**Rationale**: Memory constraints prevent resource exhaustion and ensure predictable behavior under load.

**Cross-references**: [TEST-PERF-002], [API-TEST-001]

---

### [TEST-PERF-005] Nested Performance Suites

**Scope**: Grouping related performance tests.

**Statement**: Related performance tests MAY be grouped into nested suites within `Type.Test.Performance`.

**Correct**:
```swift
extension File.System.Test.Performance {
    @Suite
    struct Throughput {
        @Test("large file write (10MB)", .timed(iterations: 5, warmup: 1, threshold: .seconds(5)))
        func largeFileWrite() throws {
            let data = [UInt8](repeating: 0, count: 10_000_000)
            try File.System.write(data, to: tempPath)
        }
    }

    @Suite
    struct AllocationTracking {
        @Test("buffer read is low-allocation", .timed(iterations: 10, maxAllocations: 256_000))
        func bufferReadLowAllocation() throws {
            _ = try File.System.read(from: tempPath)
        }
    }
}
```

**Rationale**: Nested suites enable selective execution of performance test categories.

**Cross-references**: [TEST-PERF-001]

### [TEST-PERF-006] Benchmarks as Architectural Validation

**Scope**: Using performance measurements to validate design decisions.

**Statement**: When introducing architectural complexity (new packages, layers, abstractions), benchmarks SHOULD be used to validate that the complexity provides proportional payoff. Performance improvements MUST justify their implementation cost.

**Design Validation Pattern**:

| Complexity Cost | Required Improvement | Example |
|-----------------|---------------------|---------|
| Single package, straightforward | 2x improvement sufficient | Simple optimization |
| Multiple packages, platform conditionals | 5x+ improvement needed | Platform primitives layer |
| C shims, build system overhead | 6x+ improvement needed | Native UUID parsing |

**Example: Platform Primitives Validation**

The platform primitives layer for UUID parsing adds significant complexity:
- Three separate packages (Darwin, Linux, Windows)
- Platform-conditional dependencies
- C shim headers with build system overhead

Benchmarking showed:
- Pure Swift: 9x slower than Foundation
- With native `uuid_parse`: 1.5x slower than Foundation
- **6x improvement** justifies the architectural investment

If native parsing had achieved only 2x improvement, the layer might not carry its weight.

**Correct**:
```swift
extension RFC_4122.UUID.Test.Performance {
    @Test("native parsing vs pure Swift", .timed(iterations: 1000))
    func nativeParsing() throws {
        // Measure with native path
        let nativeResult = Benchmark.measure { UUID.parse(hyphenatedString) }

        // Measure with pure Swift fallback
        let swiftResult = Benchmark.measure { UUID.parsePureSwift(hyphenatedString) }

        // Document the ratio for architectural validation
        let ratio = swiftResult.median / nativeResult.median
        // If ratio < 5x, reconsider whether the complexity is worth it
    }
}
```

**Rationale**: Architectural complexity must earn its existence through measurable benefits. Benchmarks provide objective evidence for design decisions, preventing both premature optimization and unjustified complexity.

**Cross-references**: [TEST-PERF-001], [PRIM-ARCH-001], [API-TEST-001]

## Complete Example

**Scope**: Reference implementation combining all patterns.

**Statement**: Test files SHOULD follow this structure as a template.

```swift
// File: File.Path Tests.swift

import Testing Extras  // Current: provides #TestSuites and .timed()
// import Testing      // Future: once swift-testing is finalized

@testable import File_System

extension File.Path {
    #TestSuites  // Current
    // #Tests    // Future
}

// MARK: - Unit Tests

extension File.Path.Test.Unit {
    @Test("init from string literal")
    func initFromStringLiteral() throws {
        let path: File.Path = "/usr/bin"
        #expect(path.string == "/usr/bin")
    }

    @Test("parent returns containing directory")
    func parentReturnsContainingDirectory() throws {
        let path = try File.Path("/usr/bin/swift")
        #expect(path.parent?.string == "/usr/bin")
    }
}

// MARK: - Edge Cases

extension File.Path.Test.EdgeCase {
    @Test("empty path throws error")
    func emptyPath() {
        #expect(throws: File.Path.Error.empty) {
            try File.Path("")
        }
    }
}

// MARK: - Performance

extension File.Path.Test.Performance {
    @Test("path appending", .timed(iterations: 100, warmup: 10))
    func pathAppending() {
        var path = File.Path.root
        for i in 0..<100 {
            path = path.appending("dir\(i)")
        }
    }
}
```

## Summary

| Rule ID | Pattern | Purpose |
|---------|---------|---------|
| [TEST-ORG-001] | `#TestSuites` (current) / `#Tests` (future) on type extension | Generate Unit/EdgeCase/Integration/Performance suites |
| [TEST-ORG-002] | `Type.Test.Unit` | Standard functionality tests |
| [TEST-ORG-002] | `Type.Test.EdgeCase` | Boundary conditions, error paths |
| [TEST-ORG-002] | `Type.Test.Integration` | Cross-type interactions |
| [TEST-ORG-002] | `Type.Test.Performance` | Timing and allocation tracking |
| [TEST-ORG-002] | `Type.Test.Snapshot` | Snapshot comparison tests (future) |
| [TEST-ORG-005] | Protocol tests in parent type | Protocols cannot have nested types |
| [TEST-FILE-001] | File naming `Type.Nested Tests.swift` | Mirror type hierarchy |
| [TEST-FILE-002] | Separate test targets | Organize by architectural layer |
| [TEST-PERF-002] | `.timed()` configuration | Performance measurement and thresholds |

## Framework Migration

When swift-testing is finalized, migration involves:

1. Change import from `Testing Extras` to `Testing`
2. Replace `#TestSuites` with `#Tests`
3. Add `Type.Test.Snapshot` tests if needed

The test structure and APIs are designed to be compatible, minimizing migration effort.

## Cross-Platform Verification

### [TEST-PLAT-001] Semantic Verification for Non-Compilable Targets

**Scope**: Cross-platform code changes when the target platform cannot be compiled locally.

**Statement**: When changing cross-platform APIs, non-compilable targets MUST be verified using pattern-based search when compilation is unavailable.

#### The Verification Gap

When developing cross-platform code on macOS, Linux and Windows targets cannot be compiled. `#if os(Linux)` blocks are syntactically parsed but not type-checked. Errors hide until CI runs on the actual platform.

#### The Process

When changing a cross-platform API:

1. **Change the API definition**
2. **Fix all call sites you can compile** (development platform)
3. **Search for the old API pattern** in non-compilable targets:
   ```bash
   grep -r "\.oldMethodName" Sources/
   grep -r "OldTypeName" Sources/
   ```
4. **Fix those files semantically** (reading context, applying the same pattern)
5. **Trust CI to catch what you missed**

**Correct**:
```bash
# After changing .unsafeCString to .withUnsafeCString
grep -r "\.unsafeCString" Sources/

# Found: Sources/Windows Primitives/Kernel.Path.swift:42
# Fix the Windows file even though it won't compile locally
```

**Incorrect**:
```swift
// ❌ Assuming Darwin-only code is representative
// Darwin has 0 usages, so we're done
// (Windows actually has 11 files using the old API)
```

#### Pattern Search Limitations

Pattern search substitutes for compilation imperfectly:

| Limitation | Mitigation |
|------------|------------|
| Finds string patterns, not semantic usage | Use distinctive API names |
| Aliases and indirection can be missed | Search for common patterns |
| False positives possible | Review context before changing |

For known API changes with distinctive names, pattern search provides useful confidence. CI provides the final verification.

#### Platform-Specific Code Varies

Platform-specific code varies platform-specifically. Never assume one platform's usage pattern represents others:

| Platform | `.unsafeCString` Usage |
|----------|----------------------|
| Darwin | 0 files |
| Linux | 0 files |
| Windows | 11 files |

**Rationale**: Without local compilation, the only verification is pattern-based search. This is imperfect but catches most regressions. CI on actual platforms provides the authoritative check.
